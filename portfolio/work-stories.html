Skeptical Authoring: An AI-Native Design Pattern

Context: Prototyping structured authoring workflows (DocBuilderA schema-driven writing environment prototype that structures documents as a knowledge graph.) and scaling them to end-user AI interfaces (e.g., Autodesk Forma).

The Concept

Whether designing an internal IDE for technical writers or a generative AI interface for end-users, the core challenge remains the same: How do you design the interface between human judgment and machine suggestion so that the human retains control, the output is coherent, and the system earns trust?

My approach centers on a philosophy I call "Skeptical Authoring." It treats AI not as an autonomous creator, but as a bounded proposition engine. It operates within light, schema-first scaffolding, with the human in the loop for validation and decision-making. The structure is intentionally flexible: authors can override defaults, and ambiguous material can be routed into a controlled fallback area rather than forced into premature categorization. The system's job is to surface contextually constrained suggestions; the user's job is to validate and orchestrate.

Core Principles

1. Bounded Proposal, Not Autonomous Generation

The system never writes freely. It proposes content within explicit structural constraints—a topic type, a required field set, or a defined relationship to other nodes. This is the opposite of "let the AI write it." It is: "The AI surfaces a candidate; the schema validates it; the author decides."

2. Ordered Outputs from Unordered Inputs

Real-world information arrives chaotically: Jira tickets, Slack threads, meeting notes, partial drafts, or ambiguous user prompts. The system's job is to:

  • Ingest without requiring structure upfront.
  • Classify by inferring intent, audience, and relationships.
  • Sequence into a usable order based on the user journey, not the ingestion order.

This produces machine-readable, document-map-ready output that both humans and AI systems can navigate reliably.

3. Skeptical Guidance Built Into the Workflow

Once content is consolidated into this structured environment, the interface can provide skeptical UI assistance, flagging inconsistencies, missing definitions, or misclassified topics while preserving author control. The system assumes the input (human or AI-generated) may be incomplete or misclassified. It guides the user toward coherence rather than silently accepting drift. Examples include:

  • "This looks like a procedure, but it's tagged as a concept. Reclassify?"
  • "This topic references a term not defined in the project glossary. Add definition?"

Application: From Authoring to User-Facing AI

These exact structural principles apply directly when designing content for AI-powered features that end users interact with:

Internal Environment (e.g., DocBuilder) User-Facing AI Feature (e.g., Forma)
AI proposes a document topic; author accepts/rejects. AI proposes a structural design option; user accepts/rejects.
The DITA schema enforces valid document structure. The application UI enforces valid geometric/data parameters.
Skeptical prompts surface content inconsistencies. Transparency cues explain the AI's reasoning to the user.
Chaotic inputs → Ordered documentation deliverables. Ambiguous user intent → Structured, viable project suggestions.
The Design Philosophy: When an AI suggests a design option or automates a workflow step, surface the suggestion transparently, explain the reasoning, and let the user decide. Skeptical guidance ensures coherence without forcing the user to surrender control.

Content Design: Self-Service Eligibility Tool

Role: Content Designer | Context: SEAI Retrofit Grant Application

Situation

This project was created for a community organization that needed a fast, reliable way to check retrofit grant eligibility for walk‑in visitors. The official Sustainable Energy Authority of Ireland (SEAI)The national government body responsible for delivering home energy retrofit grants. process was spread across a 12-page document written for policy experts, making it difficult for staff to give clear answers quickly.

Task

Reduce cognitive load by translating a complex government policy into a self-service, digital flow. The interface needed to make external documentation optional rather than required.

Actions

1. Mapped the User Mental Model

I worked directly with a subject‑matter expert who walked me through the underlying rules, exceptions, and the real‑world questions people asked. Each rule and exception was validated directly with the SME, keeping the human in the loop throughout. Together we iterated on a simplified flow that ignored the bureaucracy and focused on the only two things visitors actually needed to know:

  • Am I eligible for a grant?
  • If so, which grant(s)?

2. Designed the Decision-Tree UX

I collapsed the multi-page policy into a branching questionnaire. By writing contextual microcopy and defining distinct, unambiguous outcome states, I hid the complexity of the SEAI decision tree entirely. This acted as a guided onboarding flow, helping users learn the rules step-by-step.

3. Applied Progressive Disclosure

I structured the diverse grant information into easily navigable interfaces, employing tooltips to define necessary jargon in-flow without overwhelming the user or forcing them to leave the screen.

Result

  • Reduced a 12-page government policy to a 3-question eligibility flow.
  • Produced clear, unambiguous outcomes that removed the need for staff to interpret policy language.
  • Eliminated the reliance on printed guides; the flow itself became the documentation.

Product UI: Interface-Level Content Architecture

Role: Senior Technical Writer | Context: DT One API Integration Portal

Situation

DT One provided B2B API integrationsApplication Programming Interface: A set of rules that lets different software systems communicate with each other. for global airtime and data top-ups. However, external developers trying to integrate with the platform were struggling. The API reference documentation was provided as raw, unformatted Swagger files, and the conceptual integration guides were distributed as static offline PowerPoint presentations. The lack of a centralized, searchable portal made developer onboarding unnecessarily difficult.

Task

Design, build, and populate a modern, centralized Developer Portal to replace the fragmented offline assets, establishing a self-service developer experience.

Actions

1. Discovered Real User Needs via Support Staff

Rather than guessing what external developers needed, I went directly to the internal support staff who were fielding the daily integration tickets. They understood exactly where the raw Swagger files failed to explain the underlying business logic, highlighting the real customer pain points.

2. Established a Consistent Design Pattern

Working with Support, Product, and Engineering ensured content was integrated early and aligned with real integration workflows. I extracted the core business logic from the legacy files. Rather than just rewriting the text, I designed an interface-level content architecture. I built a consistent four-tab template to guide developers through the core workflow: discover products, post transactions, and retrieve status. This treated the documentation like a software interface, not a static reference. The resulting structure made the documentation both human-navigable and machine-readable, enabling clearer mapping between workflow steps and API reference material.

3. Executed Progressive Disclosure

As the sole technical writer for the project, I rewrote the integration guides, structured the API reference materials, and deployed the new architecture in three months. By linking the task guides directly to an enriched API reference, I used progressive disclosureAn interaction design technique that sequences information and actions across several screens to reduce user overwhelm. to separate high-level workflow context from raw reference data, bridging the gap between "how do I integrate airtime?" and "which endpoint do I call?".

Result

  • Launched the developer portal in a three-month timeframe using a repeatable, scalable design pattern.
  • Eliminated reliance on offline PDFs and PowerPoints.
  • Reduced friction for external teams integrating with the DT One platform by replacing static endpoint lists with task-oriented workflows.

ENM Case Study — Product Description Governance

Role: Technical Writer | Context: Ericsson Network Manager (ENM)A centralized management system used by telecom operators to configure and monitor radio networks. DITA migration

Situation

The ENM Product Description had been expanded by more than fifty teams over several years. It no longer reflected the product or its information architectureThe structural design of shared information environments, helping users find exactly what they're looking for.. Terminology drifted, procedural content was mixed into descriptive chapters, and the document structure did not match what users saw on the screen.

A specific problem was the hardware installation content. It existed as a single, screenshot‑heavy Confluence page embedded within the Product Description. The fundamental issue was structural: a user looking for step-by-step hardware instructions is unlikely to search inside a descriptive guide. Furthermore, the format was brittle to edit and unsuitable for a direct DITA conversionTranslating flat, unstructured documents into the DITA XML standard for modular reuse..

Task

My assignment was to execute the DITA conversion for this specific guide. I quickly identified that a literal conversion would preserve all the structural flaws.

I needed to rewrite the Product Description into a coherent, UI‑aligned structure. This required extracting the hardware content into a standalone book, relocating misplaced procedures, and managing reviews from fifty contributing teams without delaying the main software release.

Actions

1. Built an alliance to validate the rewrite

As a new hire, I did not have the authority to halt a scheduled conversion to force an architectural rewrite. I raised the issue with my Scrum Master, who recognized the scale of the problem and connected me with the Lead Architect. The architect had the seniority to confirm that a deeper structural overhaul was necessary.

2. Secured resources for an architectural split

I proposed extracting the unmaintainable hardware monolith into a standalone DITA guide. Management initially resisted because document counts were tied to departmental headcount budgets; assigning me to write a completely new guide meant removing me from other tasks and creating a resourcing gap.

I demonstrated that hiding hardware configuration inside a descriptive document guaranteed users would miss it, while creating severe maintenance overhead for the writing team. Management conceded, reassigning my other deliverables so I could dedicate the necessary time to split the guides.

3. Rebuilt the information architecture

I remapped the entire Product Description to mirror the product UI, reordering legacy content so that each section matched how users navigated the software.

To prevent drift during ENM’s rapid release cadence, I applied a strict triage model to procedural content: if a procedure existed elsewhere, I removed the duplicate. If it was valid but orphaned, I coordinated with the owning team to integrate it into their guide.

This created a scalable structure that reduced drift and supported consistent updates across 50+ contributing teams.

4. Ran an architect‑validated review workflow

I could not enforce deadlines across fifty teams alone. With the Lead Architect as my co‑sponsor, we established a predictable review workflow. I drafted each section, he reviewed and approved it, and only then was it sent to the contributing teams.

Teams were given clear deadlines. We established a firm rule: if we received no feedback by the deadline, the architect‑approved content was published. Silence was treated as approval.

Result

  • Delivered a rewritten Product Description that matched the UI in a single launch, causing zero delays to ENM’s release train.
  • Successfully justified resource reallocation by proving the usability and maintenance risks of the legacy structure to management.
  • Replaced an embedded monolith with a standalone Hardware Installation Guide organized by server layout.
  • Secured compliance from fifty stakeholder teams using a strict, architect‑validated review workflow.

SAP Case Study — UI Strings and Product Influence

Role: Senior Information Developer | Context: SAP Master Data Governance (MDG)An enterprise data management application used to centrally govern critical business records. Foundation

Situation

During the development of SAP MDG, the delivery of a UI screenUser Interface: The visual elements (buttons, menus, screens) a user interacts with in a software application. stalled due to a terminology deadlock. A developer had created a UI checkbox label containing five words of technical jargon: "Entity Attribute Change Relevance Indicator."

The UI Designer blocked it for being unreadable. However, the Product Owner defended the label by explaining the underlying database schemasThe skeleton structure that represents the logical view of the entire database. it mapped to. The team was engaged in a circular argument, resulting in a blocked feature and an impossible concept to document for end-users.

Task

Stop the debate between design and engineering, and facilitate a terminology decision based on business intent rather than database architecture.

Actions

1. Halted the Technical Debate

I intervened to stop the technical debate about database schemas. I asked the Product Owner a direct question focused entirely on the user journey: "Forget the database. If a business user checks this box, what actually happens to the process?"

2. Translated Database Logic into Business Intent

The Product Owner paused, shifted away from the technical specifications, and admitted: "Basically... you decide if the field is governed by the Change Request or not." This was a human-in-the-loop clarification process, grounded in direct questioning and collaborative reasoning.

3. Executed the Simplification

Using his own explanation, I proposed stripping away the technical jargon entirely. I changed the checkbox label simply to "Governed" and defined the overarching user task as "Define Governance Scope."

4. Codified the Standard

To ensure this wasn't just a one-off fix, I immediately entered "Governance Scope" into SAPTerm, formally codifying it as the standard taxonomy for the entire product suite.

Result

  • Unblocked the UI design phase by replacing a five-word database string with a single, user-focused word.
  • Simplified the corresponding end-user documentation before it was even drafted.
  • Established a strong partnership with the Product Owner, proving that Technical Writers serve as strategic design partners who actively shape the software. The Product Owner later endorsed this exact interaction:
    "Our conversations made the product better as he always had suggestions how to frame the problem better, how to 'name' the components, how the UI should work etc. He brought the logic and the language integrity into the SAP documentation."
