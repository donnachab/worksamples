Ericsson ENM: Reworking a Product Description with structural and content issues

Role: Technical Writer

Context: ENM (Ericsson Network Manager) — Confluence → Ixiasoft DITA migration

Situation

I was asked to migrate the ENM Product Description into DITA. The document had been built over several years by more than fifty teams, each writing in different styles and with different assumptions. The result was uneven.

  • Terminology drifted across chapters.
  • The order of titles did not match the UI.
  • Procedural content appeared in what was meant to be a descriptive document.
  • A hardware installation guide had been embedded as a single, oversized Confluence page with diagrams but no explanation of when or why to use each configuration.

Task

Produce a Product Description that was structurally coherent, technically accurate, aligned with the UI, and ready for DITA, while passing review by fifty contributing teams without blocking the release.

Actions

1. Reframed the assignment

I explained that a mechanical DITA conversion would not solve the underlying problems. With the Lead Architect’s support, the work was re‑scoped as a full structural rewrite.

2. Analysed the existing document

  • Reviewed the document end‑to‑end.
  • Mapped chapters to the UI.
  • Identified terminology inconsistencies.
  • Flagged procedural content.
  • Documented gaps and missing context.

3. Split out the hardware installation guide

The embedded hardware content was inappropriate for a Product Description and lacked essential context. I proposed creating a separate Hardware Installation Guide.

This required negotiation because document count was tied to headcount allocation. The manager initially resisted adding another deliverable. I requested that several other guides assigned to me be reassigned, which was accepted.

4. Rebuilt the Product Description in UI order

  • Rewrote the document to follow the UI.
  • Aligned terminology across teams.
  • Established a consistent pattern for describing features.
  • Removed procedural content.
  • Ensured each chapter reflected actual product behaviour.

5. Prepared the content for DITA

  • Broke content into DITA‑appropriate topics.
  • Applied correct topic types.
  • Separated concept, task, and reference.
  • Created a maintainable structure for future releases.

6. Ran a staggered review and publication cycle

  • Architect reviewed each rewritten section first.
  • Released content in small batches.
  • Each batch had its own review window.
  • Silence counted as approval.
  • Resolved objections before releasing the next batch.
  • Published and migrated content after all batches cleared review.

7. Managed incoming tickets during the rewrite

  • Recognised that many tickets were effectively ticket bankruptcy.
  • Reviewed each ticket for useful insight.
  • Integrated valuable findings into the rewrite.
  • Deferred fixes for content that was being replaced.

Result

  • A Product Description aligned with the UI and consistent in terminology.
  • Clear separation between descriptive and procedural content.
  • A standalone Hardware Installation Guide with proper context.
  • A staggered review model that avoided bottlenecks.
  • A maintainable DITA structure for future releases.
  • A realistic workload distribution after negotiating document ownership.
  • A technically accurate, coherent document produced with the Lead Architect.
