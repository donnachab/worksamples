DT One Developer Portal

Role: Technical Writer and Information Architect

I replaced unsearchable PowerPoint presentations and bare-bones Swagger specifications with searchable, task-based guides. The existing Swagger listed what endpoints existed, but not how to use them for specific integrations.

I designed a consistent four-tab template to guide developers through the core workflow: discover products, post transactions, and retrieve status. I also enriched the Application Programming InterfaceA set of functions that allows software applications to talk to each other, used here for communication between systems. (API) reference with context and examples, linking it directly to the task guides. This bridged the gap between "how do I integrate airtime?" and "which endpoint do I call?".

The result was accurate, searchable developer documentation that replaced static files and provided a clear path to successful integration.

Analysis: Why This Design Works

Design Principles

  • Task-oriented structure: Matches developer mental models ("I need to integrate airtime") rather than the endpoint-focused API structure.
  • Sequential workflow: Leads developers step-by-step from product discovery to transaction completion.
  • Prominent warnings: Highlights required fields and common mistakes to address top support issues.
  • Visual clarity: Uses Mermaid diagrams to make complex transaction states understandable.
  • Scannable content: Uses accordions so developers can expand only the details they need.
  • Consistent pattern: Applies one structure across 7+ product types to reduce the learning curve.

Impact

  • Scalability: Successfully applied across mobile, eSIM, and gift card products.
  • Reduced support burden: Addressed the top two categories of support questions directly in the docs.
  • Faster integration: Gave developers a clear path from intent to implementation.
